# Ripple UI/UX Constitution & Design Board Guidelines

This document governs the design system, multi-project userure, and usability constraints of the Ripple Dashboard. Any future changes, screen redesigns, or feature integrations must justify themselves against these laws.

---

## 🏛️ UI/UX Design Board Principles

Ripple is designed for developers who value speed, clarity, and precision. It does not use heavy front-end frameworks or bloated libraries. Its interface must be filtered through the Design Advisory Board lenses:

### 1. Dieter Rams — Essentialism ("Less but Better")
*   **The Constraint:** Remove everything until removing hurts usefulness. 
*   **Application:** Zero promotional widgets, decorative animations, or redundant navigation. Keep controls minimal. The dashboard is a single-screen command center.
*   **Theme:** Default to a high-end, premium dark mode (`#0d1117` background with `#161b22` cards) to reduce eye strain and look exceptionally modern.

### 2. Edward Tufte — Data Density & Signal-to-Noise
*   **The Constraint:** Maximize the information ratio. Every pixel must carry data.
*   **Application:** Use high-density CSS grids. Display the before/after session diff stats immediately next to each git commit. Use micro SVG sparklines for traffic trends rather than loading a massive Chart.js/D3 library.
*   **Typography:** Use a clean monospace font (like *JetBrains Mono* or *Fira Code*) for commit hashes, metrics, and file lists to maintain technical legibility.

### 3. Steve Krug — Usability & Friction ("Don't Make Me Think")
*   **The Constraint:** A stranger must understand the state of the system in 5 seconds.
*   **Application:** The primary layout is split into three panels that answer three immediate questions:
    1.  *Where are we?* (Active Project name, live session metrics, goal completion percentages).
    2.  *What happened?* (Git commit history correlated with traffic/engagement shifts).
    3.  *What should I do?* (Actionable suggestions generated by the intelligence layer).

### 4. Johannes Itten — Color Contrast
*   **The Constraint:** A limited palette with saturated colors reserved strictly for primary actions or critical status signals.
*   **Application:**
    *   **Backgrounds:** Deep charcoal and dark gray (`#0d0e12`, `#1a1d24`).
    *   **Text:** Crisp white (`#ffffff`) for titles, muted gray (`#8b949e`) for context.
    *   **Status Indicators:** Neon Green (`#2ea44f`) for positive impact, Neon Amber (`#d29922`) for warnings/new deployments, Neon Red (`#f85149`) for traffic drops.

## 🔄 Multi-Project userure & Configuration

Ripple is natively aware of multi-project/tenant environments. Multiple projects can be tracked on the same machine and managed from a single dashboard.

### 1. How to Add a Project
Open `ripple.config.json` and append a new project object to the `"projects"` array:
```json
{
  "key": "new_project_key",
  "name": "My New Project",
  "url": "https://newproject.com",
  "sessions_dir": "C:\\path\\to\\project\\sessions",
  "git_repo": "C:\\path\\to\\project\\git_repo",
  "goals": [
    "Increase checkout completion rate",
    "Reduce bounce rate on registration page"
  ],
  "interaction_events": [
    "checkout_clicked",
    "registration_submitted"
  ]
}
```

### 2. How to Edit / Configure a Project
*   **Modify Goals:** Edit the `"goals"` array. Ripple supports two types of goals:
    *   *Milestone Goals*: Simple plain-text checklist tasks (e.g., `"Add warning dismiss audio trigger"`). Ripple's backend evaluates these by searching recent git commit messages.
    *   *Metric Goals*: Structured rule-based thresholds with the prefix `[Metric]` (e.g., `[Metric] view:swag-store duration < 15s` or `[Metric] event:order_clicked conversion > 10%`). Ripple calculates actual sessional telemetry values (view durations and event conversion rates) within a configurable lookback window (default 7 days) and automatically checks them off or raises alerts.
*   **Modify Lookback Window**: Edit `lookback_days` (per project or globally) to configure the moving average evaluation window for metrics.
*   **Update Interaction Events:** Edit the `"interaction_events"` array to capture custom event triggers sent by `ripple.js` (e.g. tracking registration, play, purchase).

### 3. How to Select & Load Projects in the Dashboard
*   **Dynamic Loader:** The dashboard UI reads the generated `data/project_analytics.json` containing the data for all projects.
*   **Sidebar Navigation:** A sidebar dropdown or list displays all active project names.
*   **Project Switching:** Selecting a project switches the active context:
    *   All cards, traffic metrics, and funnels reload instantly.
    *   The git deployment list is repopulated with that project's commit history.
    *   The intelligence panel swaps to display suggestions matching the selected project's keys.

## 📋 Standard Multi-Project Setup Instructions (User Guide)

When running Ripple, follow this checklist to manage multiple sites:

1.  **Add Tracker Code:** Place the tracking script on all sites:
    ```html
    <script src="ripple.js" data-project="project_key"></script>
    ```
2.  **Declare in Config:** Add each project key, folders, and goals in `ripple.config.json`.
3.  **Run Compilation:** Execute the analyzer script to build the compiled data for all sites:
    ```bash
    python scripts/analyze.py
    ```
4.  **Open Dashboard:** Serve the dashboard locally:
    ```bash
    # Run a simple local server
    php -S localhost:8000
    ```
    Navigate to `http://localhost:8000/src/dashboard/` and select the desired project from the dropdown.
